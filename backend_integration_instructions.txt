MODEL 1 — BACKEND INTEGRATION INSTRUCTIONS
==========================================


WHAT THE MODEL PRODUCES
-----------------------
The pipeline outputs one file: artifacts/model1/player_clusters.parquet
Every row is a player-season.

Columns you need:

  player_id               — join key to your players table
  player_name             — display name
  season                  — e.g. "2022" or "2015/2016"
  primary_style           — e.g. "Deep Playmaker"
  secondary_style         — e.g. "Box-to-Box" — empty string if none
  spatial_cluster_name    — e.g. "Left Wide Defender"
  final_player_archetype  — combined display label e.g. "Deep Playmaker · Box-to-Box · Left Wide Defender"
  probability_vector      — JSON string of style probabilities per cluster (use for confidence breakdown UI)
  xg_90, xa_90, shots_90, goals_90, passes_90, pass_accuracy,
  carries_90, touches_90, progressive_actions_90,
  tackles_90, interceptions_90, pressures_90
                          — the 12 per-90 stats used for styling


HOW TO SERVE IT
---------------
The parquet is pre-computed. Do NOT re-run the model on request.
Two options:

  Option 1 — Load parquet at startup
    Read it into memory once when the API starts, query it in-process.
    Fast, zero DB overhead.

  Option 2 — Write to a DB table
    After each pipeline run:  df.to_sql('player_clusters', engine)
    Serve from there. Better if you want SQL joins with other tables.


ENDPOINTS YOU WILL LIKELY NEED
-------------------------------
  GET /players/{player_id}/profile     — return that player's rows across all seasons
  GET /players/search?name=Salah       — fuzzy name filter
  GET /clusters/spatial?zone=Left+Wide+Defender   — all players in a given zone
  GET /clusters/style?style=Deep+Playmaker        — all players of a given primary style


WHEN DOES THE DATA UPDATE
--------------------------
The pipeline is run manually:
  python -m models.model1_player_clustering

Each run overwrites the parquet and the report at artifacts/model1/model1_report.txt
After each pipeline run, reload the parquet (or re-ingest to DB).
There is no live inference — everything is batch.


GOTCHA — SEASON FORMAT
-----------------------
The season column is a string but the format is inconsistent:
  some rows:  "2022"
  some rows:  "2015/2016"
Always treat season as a string. Never cast to int.
Filter with exact string match:  WHERE season = '2022'

GOTCHAS
---------
1. FEATURE ORDER MUST MATCH TRAINING ORDER
   scaler.transform() / gbr.predict() expect columns in the exact order the
   scaler was fit on. v1's order is fixed by models/model2_team_cohesion.py's
   MODEL_FEATURES list (hardcoded twice -- once there, once duplicated in
   api_server.py's /api/team-cohesion handler. Keep both in sync). v2's order
   is NOT hardcoded anywhere in the backend; it must be read from
   feature_columns_postmatch.json at startup, because it can change if the
   model is re-tuned/re-run.

2. THIS IS AN EXPLANATORY MODEL, NOT A FORECAST
   Both v1 and v2 use this match's own team_xg/team_xga (and v2 also uses
   this match's own pass-network features). They answer "given the chances
   created, how much did cohesion explain the result" -- not "what will
   happen before kickoff." Do not label predicted_goals_per_match in the UI
   as a pre-match prediction. A genuine pre-match variant was built and
   evaluated (rolling 5-match team history only) and explained far less
   variance (R2~0.08-0.11 vs ~0.34-0.35) -- it was intentionally not shipped.

3. v2's graph_features_optimized.parquet is NOT a drop-in replacement file
   name for graph_features.parquet -- they have different schemas (24 vs 12
   raw graph columns). Don't point old code at the new file or vice versa.

4. NaN HANDLING
   Both pipelines fillna(0) before scaling/predicting. If you add a new
   team/match with missing graph features (e.g. a match with zero passing
   data), it will silently predict from zeros rather than erroring -- treat
   a near-zero feature vector as a signal to fall back to the dashboard's
   existing fallback KPI block rather than trusting the prediction.

5. SCALE OF THE NUMBERS
   R2~0.34-0.35 means the model explains about a third of goal variance.
   MAE is ~0.78-0.79 goals. Don't present predicted_goals_per_match with
   false precision (e.g. "2.347 goals") -- round to 1-2 decimals and treat
   it as directional, not exact.

  artifacts/model3/scaler_prematch.pkl
  artifacts/model3/feature_columns_prematch.json

v1 and v2 are BOTH "post-appearance" models: they use THIS match's own

before kickoff (age, prior workload/rest, injury history, announced lineup
position) -- it is the only one of the three that can honestly be labelled
a pre-match prediction in the UI.

------------------------------------------------
artifacts/model3/features.parquet -- one row per (player_id, match_id).

xgb.pkl/rf.pkl + scaler.pkl expect EXACTLY this 13-feature vector, in this
order (see api_server.py's /api/injury-risk handler, which hardcodes the
same FEATURES list -- keep both copies in sync if you ever change
models/model3_injury_risk.py's FEATURES):

  minutes_played, matches_last_30_days, minutes_last_30_days,
  days_since_last_injury, age_at_match, sub_minute_flag,
  xg, xa, pressures, tackles, carry_distance, interceptions, clearances


HOW TO SERVE IT (v1)
-----------------------
Already implemented in api_server.py:
  - xgb.pkl, rf.pkl, scaler.pkl are loaded once at startup into the
    _A["m3"] dict (see _load_all_artifacts()).
  - GET /api/injury-risk?team_id=N queries the last 3+ matches per player
    on that team, averages the 13 features per player, scales them, and
    calls model.predict_proba() for risk_score.
  - risk_score is bucketed into high (>=0.67) / medium (0.4-0.67) / low
    (<0.4) inside _injury_response() -- these cutoffs are HARDCODED for
    v1's probability distribution. See GOTCHA 5 before reusing them for v3.
Do NOT re-run the model per request -- it's a batch artifact, same pattern
as Model 1/Model 2.


HOW TO UPGRADE TO v3 (optional -- not done yet, and not a drop-in swap)
--------------------------------------------------------------------------
This is NOT like the Model 2 v1->v2 upgrade (same query, swap two .pkl
files). v3's feature set is almost entirely different from v1's, so the
SQL query inside /api/injury-risk has to be rewritten, not just the model.

  1. Load the new artifacts at startup:
       prematch_model = _load(a / "model3" / "model3_prematch_best.pkl")
       prematch_prep  = _load(a / "model3" / "scaler_prematch.pkl")   # a
                         # fitted ColumnTransformer (scaling + one-hot), not
                         # a plain StandardScaler -- see GOTCHA 1.
       with open(a / "model3" / "feature_columns_prematch.json") as f:
           PREMATCH_FEATURES = json.load(f)   # 15 features, EXACT order
       with open(a / "model3" / "thresholds_prematch.json") as f:
           PREMATCH_TIERS = json.load(f)      # Low/Medium/High/Very High

  2. PREMATCH_FEATURES is:
       matches_last_30_days, age_at_match, matches_last_7_days,
       matches_last_14_days, consecutive_matches_played,
       days_rest_before_match, days_since_last_injury_capped,
       starts_last_5_matches, avg_minutes_last_5, max_minutes_last_5,
       consecutive_starts, recent_substitute_pattern,
       previous_injury_count, position_group, injury_recency_bucket
     None of these are simple AVG()-over-recent-matches like v1's query --
     they are point-in-time, strictly-prior-to-this-match rolling/streak/
     history features computed per player. You cannot compute them with a
     one-line SQL AVG(); the logic lives in
     models/model3_injury_risk.py's _add_streak_and_rest(),
     _add_rolling_workload(), _add_count_window_features(),
     _add_injury_history_features() and _map_position_group(). Either port
     that logic into the API's query layer, or (simpler) have the training
     pipeline write a per-player "current state" snapshot table/parquet
     after each run that the API just looks up by player_id.

  3. position_group and injury_recency_bucket are STRINGS (categorical),
     not numbers. scaler_prematch.pkl's ColumnTransformer one-hot-encodes
     them internally -- pass the raw string values straight into
     prematch_prep.transform(), do not encode them yourself first.

  4. Call prematch_prep.transform(raw_df[PREMATCH_FEATURES]) then
     prematch_model.predict_proba(...)[:, 1] -- two steps, same shape as v1
     (transform, then predict), just with a richer transformer.

  5. Keep v1/v2 artifacts in place even after upgrading the live endpoint.

GOTCHAS
---------
1. scaler_prematch.pkl IS NOT A PLAIN SCALER
   v1's scaler.pkl is a StandardScaler. v3's scaler_prematch.pkl is a fitted
   sklearn ColumnTransformer (StandardScaler on the numeric columns +
   OneHotEncoder on position_group/injury_recency_bucket). Calling
   .transform() works the same way, but don't assume its output width
   equals len(PREMATCH_FEATURES) -- one-hot encoding expands the two
   categorical columns into several columns each.

2. THIS IS A REAL PRE-MATCH MODEL -- BUT v1/v2 ARE NOT
   v1 and v2 both use THIS match's own minutes_played/xg/xa/pressures/
   tackles/etc, so they describe "given how this match went" -- not a
   forecast. Only v3 (prematch) excludes all of that and can honestly be
   shown as a pre-kickoff prediction. Don't relabel v1/v2's output as a
   forecast in the UI.

3. FEATURE ORDER MUST MATCH TRAINING ORDER
   Same rule as Model 2: v1's order is fixed by FEATURES in
   models/model3_injury_risk.py (hardcoded twice -- once there, once in
   api_server.py's /api/injury-risk handler; keep both in sync). v3's order
   must be read from feature_columns_prematch.json at startup, never
   hand-typed, because it can change if the model is re-tuned/re-run.

4. NaN HANDLING
   All three pipelines fillna(0) numeric columns (and "unknown" for v3's
   categorical columns) before scaling/predicting. A player with no
   workload/injury history yet will silently predict from a near-default
   feature vector rather than erroring -- treat that as a signal to flag
   the prediction as low-confidence rather than trusting it outright.

5. v1's RISK BUCKETS (high>=0.67) WILL NOT WORK FOR v3 -- DO NOT REUSE THEM
   v3's predicted probabilities top out around ~0.72 and its OWN data-driven
   "Very High" tier starts at 0.249 (see thresholds_prematch.json), not
   0.67. If you swap in the v3 model but keep _injury_response()'s
   hardcoded high/medium/low cutoffs, almost no player will ever be flagged
   "high risk" -- the cutoffs were tuned to v1's distribution, not v3's.
   Use thresholds_prematch.json's Low/Medium/High/Very High score_low/
   score_high bounds instead, and show the observed_injury_rate it stores
   per tier in the UI -- that's the genuinely useful number (e.g.
   "Very High risk players are actually injured within 30 days ~32% of the
   time in our data" is more honest and more useful than a bare 0-1 score).

6. SCALE OF THE NUMBERS -- AUC-PR, NOT JUST AUC-ROC
   v3's AUC-ROC (~0.68-0.72) looks decent but is misleading at this ~10%
   positive rate -- it's dominated by how well the large majority of
   non-injured rows rank against each other. AUC-PR (~0.10-0.22 depending
   on the split) is the honest number: a baseline that always predicts "no
   injury" gets AUC-PR equal to the positive rate (~0.10), so the model is
   meaningfully but not dramatically better than that. Don't present
   risk_score as a confident probability of injury -- present it as a
   relative ranking/tier, which is what the risk tiers in GOTCHA 5 are for.

HOW TO SERVE IT (v1)
-----------------------

    called here. Don't describe the in-game curve in the UI as coming from

as Models 1-3.

process) any time player_match_stats/matches change -- e.g. after a fresh
`python main.py` ingestion run.

There is no live inference -- everything is batch, same as Models 1-3.


GOTCHAS
---------
1. v2 NEEDS AN OPPONENT -- THE CURRENT ENDPOINT DOESN'T COLLECT ONE
   /api/win-probability's signature is win_probability(team_id: int) --
   there is no opponent parameter anywhere in the request. 17 of v2's 26
   features are opponent-relative (every *_opp column, plus elo_diff and
   season_points_per_game's partner opp_season_points_per_game). If you
   wire in v2 without adding an opponent_team_id parameter, you would have
   to zero-fill or guess every opponent feature, which throws away most of
   why v2 outperforms v1 (see GOTCHA 3). This is the central design change
   needed before v2 is worth deploying -- it is not just a file swap like
   Model 2's v1->v2 upgrade was.

2. elo_pre/opp_elo_pre/elo_diff ARE NOT A SIMPLE SQL LOOKUP
   Unlike every other v2 feature (which is some AVG()/SUM() over recent
   matches, computable per-request), Elo is a STATEFUL, ITERATIVE rating:
   each team's current Elo depends on the full chronological sequence of
   every match it has ever played in this dataset, updated one match at a
   time (see _compute_elo_ratings() in models/model5_win_probability.py).
   You cannot reconstruct a team's current rating from a single SQL query.
   Two options for serving this live:
     a) Call _compute_elo_ratings(conn) at request time and take the last
        row for each team_id -- correct, but recomputes the full match
        history (all ~2000+ matches) on every request. Fine for batch,
        too slow for a hot endpoint at scale.
     b) (Recommended) Have the training pipeline write a small
        "team_id -> current_elo" snapshot (table or JSON) after each run,
        and have the API look that up instead of recomputing from scratch.
        This file does not exist yet -- you would need to add it.
   K=20, home-advantage=100 elo points convention -- if you build a live
   updater instead of just snapshotting, replicate the exact same K/HFA/
   goal-diff-multiplier constants or ratings will silently diverge from
   what the model was trained on.

3. FEATURE ORDER MUST MATCH TRAINING ORDER
   Same rule as Models 2/3: v1's order is fixed by FEATURES_PRE_MATCH in
   models/model5_win_probability.py (hardcoded in TWO places in
   api_server.py -- keep both in sync). v2's order must be read from
   feature_columns_pre_optimized.json at startup, never hand-typed.

4. NaN HANDLING
   Both pipelines fillna(0) before scaling/predicting. A team with no
   match history yet (or, for v2, an opponent with no history) will
   silently predict from a near-zero feature vector rather than erroring
   -- treat that as a signal to fall back to v1 or the dashboard's
   existing fallback formula rather than trusting the prediction.

5. THE IN-GAME CURVE IN THE CURRENT RESPONSE IS NOT A MODEL PREDICTION
   See "HOW TO SERVE IT" above -- the per-minute series in the existing
   /api/win-probability response is a hardcoded formula, not Model 5B's
   output. Wiring up the real in-game model (gbc_ingame.pkl / v2's
   gbc_ingame_optimized.pkl) is a separate task from this one.

6. SCALE OF THE NUMBERS
   v2's held-out-2022 accuracy is ~0.46 against a naive (always-predict-
   majority-class) baseline of ~0.37 -- a real but modest lift, not a
   confident predictor. The draw class in particular is almost never
   correctly predicted on held-out data (see the confusion matrix in
   model5_optimized_diagnostics.txt) -- this is a well-known limitation in
   football outcome forecasting generally, not a bug. Present win/draw/
   loss percentages as directional, not as confident probabilities, and
   don't be surprised if "draw" stays under-represented in predictions
   even when it's the right call.
